Напрям класифікації Реалізація в системі Пояснення
Управління вимогами Таблиця покриття вимог реалізацією Система показує ID вимоги, її формулювання, реалізацію, доказ і статус виконання.
Управління бронюваннями Адміністративна панель Адміністратор може переглядати заявки, підтверджувати, заселяти, скасовувати або видаляти бронювання.
Технології інтерфейсу HTML, CSS, JavaScript HTML формує структуру сторінок, CSS відповідає за оформлення, JavaScript реалізує логіку роботи.
Технологія комунікації Fetch API та HTTP-запити Клієнтська частина передає запити до серверної частини для отримання, створення, оновлення та видалення даних.
Серверна частина Node.js + Express Сервер обробляє API-запити та забезпечує роботу з даними номерів і бронювань.
Технологія збереження даних JSON-файли на серверній стороні Дані про номери та бронювання зберігаються у файлах даних, а не лише в інтерфейсі користувача.
Комунікація між сторінками Гіперпосилання та спільний script.js Користувач переходить між сторінками, а спільний JavaScript-файл забезпечує роботу логіки системи.

Класифікація за ступенем автоматизації

У таблиці показано, які процеси в системі виконуються автоматично, а які потребують участі користувача або адміністратора.

Процес Ступінь автоматизації Пояснення
Відображення номерів Автоматизовано Система автоматично виводить картки номерів на головній сторінці.
Вибір номера Частково автоматизовано Користувач обирає номер, а система передає вибір на сторінку бронювання.
Створення бронювання Автоматизовано після введення даних Після заповнення форми система створює заявку та зберігає її.
Перевірка обов’язкових полів Автоматизовано Система перевіряє ім’я клієнта, телефон, дати та вибір номера.
Керування заявками Частково автоматизовано Адміністратор натискає кнопку дії, а система автоматично змінює статус заявки.
Видалення бронювання Частково автоматизовано Адміністратор запускає дію видалення, а система прибирає заявку та оновлює стан номера.
Підрахунок статистики Автоматизовано Система автоматично показує кількість номерів, бронювань, вільних номерів і суму.

Користувацький рівень Описує потреби клієнта й адміністратора Клієнт переглядає номери та створює бронювання, адміністратор керує заявками Покрито сторінками index.html, booking.html та admin.html
Функціональний рівень Описує конкретні функції системи Перегляд номерів, створення бронювання, зміна статусів, видалення заявки Покрито функціями інтерфейсу та логікою script.js

Нефункціональний рівень Описує якість роботи системи та зручність використання Український інтерфейс, зрозуміла навігація, повідомлення про результат дії Покрито оформленням сторінок, підписами, кнопками та повідомленнями
Рівень реалізації Показує, якими технічними засобами реалізовано вимоги HTML, CSS, JavaScript, Node.js, Express, JSON-файли Покрито клієнтською та серверною частинами системи

Оцінка повноти вимог

Повнота вимог означає, що для основних частин системи визначено потрібні функції, дані, ролі користувачів, способи керування та підтвердження реалізації.

Критерій повноти Що перевіряється Стан у проєкті «Готель»
Наявність ролей користувачів Чи визначено, хто працює із системою Покрито: є клієнт і адміністратор
Наявність основних функцій Чи описано головні дії системи Покрито: перегляд номерів, створення бронювання, керування заявками
Наявність вимог до даних Чи визначено, які дані потрібні для бронювання Покрито: клієнт, телефон, номер, дати, статус і коментар
Наявність вимог до інтерфейсу Чи зрозуміло, як користувач взаємодіє із системою Покрито: меню, кнопки, форми, таблиці, статуси
Наявність вимог до збереження Чи зберігаються створені заявки Покрито: дані зберігаються через серверну частину у JSON-файлах
Наявність підтвердження реалізації Чи можна довести, що вимоги реалізовано Покрито: є таблиця покриття вимог і скріни роботи системи
Повнота покриття Чи всі основні модулі системи мають відповідні вимоги Покрито: головна сторінка, бронювання, адмін-панель, вимоги, серверна частина

Впорядкованість вимог за важливістю і стабільністю

У цьому блоці показано, які вимоги програмної системи «Готель» є найбільш важливими для роботи системи, а також наскільки вони стабільні під час подальшого розвитку проєкту.

ID Вимога Важливість Стабільність Обґрунтування
FR-01 Система повинна відображати список номерів готелю Висока Стабільна Без перегляду номерів користувач не зможе обрати номер для бронювання.
FR-02 Клієнт повинен мати можливість створити бронювання Висока Стабільна Це головна функція системи, без якої проєкт не виконує своє призначення.
FR-03 Система повинна перевіряти обов’язкові поля Висока Стабільна Перевірка полів потрібна для коректності даних і запобігання помилковим бронюванням.
FR-04 Система повинна зберігати бронювання Висока Стабільна Збереження заявки необхідне для подальшої роботи адміністратора.
FR-05 Адміністратор повинен бачити список заявок Висока Стабільна Адміністратор не зможе керувати бронюваннями без таблиці заявок.
FR-06 Адміністратор повинен керувати бронюваннями Висока Стабільна Підтвердження, заселення, скасування та видалення є основними діями адміністратора.
FR-07 Кожен номер повинен мати статус Висока Стабільна Статуси потрібні для узгодженості між номерами, заявками та діями адміністратора.
NFR-01 Інтерфейс повинен бути українською мовою Середня Стабільна Ця вимога важлива для зручності використання в навчальному проєкті, але не змінює основну бізнес-логіку.
DR-01 Бронювання повинно містити клієнта, номер і дати проживання Висока Стабільна Ці дані є обов’язковими для створення повноцінної заявки на проживання.
UI-01 Система повинна мати зрозумілу навігацію між сторінками Середня Відносно стабільна Навігація потрібна для зручності, але її зовнішній вигляд може змінюватися під час розвитку інтерфейсу.
OPT-01 Додаткові візуальні блоки, фото та декоративні елементи Низька Може змінюватися Ці елементи покращують вигляд системи, але не є критичними для бронювання та адміністрування.
