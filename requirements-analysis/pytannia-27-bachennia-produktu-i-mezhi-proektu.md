# Питання 27. Бачення продукту і межі проєкту

## Питання

**Бачення продукту і межі проєкту.**

## Відповідь

Бачення продукту — це загальне уявлення про те, для чого створюється програмна система, яку проблему вона вирішує, хто буде її використовувати та яку користь вона повинна дати користувачам.

Межі проєкту визначають, що входить до складу системи, які функції реалізуються, які ролі користувачів підтримуються, а які можливості залишаються поза межами поточної версії. Межі проєкту потрібні для того, щоб не розширювати систему хаотично й чітко розуміти, що саме повинно бути реалізовано.

У проєкті **«Програмна система “Готель”»** бачення продукту полягає у створенні вебзастосунку для перегляду номерів готелю, створення бронювань, обліку заявок клієнтів, керування номерами та демонстрації покриття вимог реалізацією.

## Бачення продукту

Програмна система **«Готель»** призначена для спрощення роботи з бронюваннями номерів. Користувач може переглянути доступні номери, ознайомитися з їх типом, ціною, описом і статусом, після чого створити бронювання.

Адміністратор, у свою чергу, може переглядати створені заявки, керувати статусами бронювань, підтверджувати, заселяти, скасовувати або видаляти заявки. Також у системі додано можливість керування номерним фондом: адміністратор може додавати номер і редагувати інформацію про наявні номери.

Основна ідея продукту полягає в тому, щоб об’єднати в одному навчальному вебзастосунку головні процеси роботи готелю:

* перегляд номерів;
* створення бронювання;
* збереження заявки;
* перегляд бронювань адміністратором;
* керування статусами заявок;
* керування номерами;
* перегляд вимог;
* використання глосарію;
* підтвердження реалізації вимог через скріни та таблицю покриття.

Таким чином, система **«Готель»** є не просто статичним сайтом, а навчальною програмною системою, яка демонструє зв’язок між вимогами, інтерфейсом, сценаріями користувачів і реальною поведінкою програми.

## Цільові користувачі продукту

У системі визначено дві основні ролі користувачів:

| Користувач    | Основна мета                                                                         |
| ------------- | ------------------------------------------------------------------------------------ |
| Клієнт        | Переглянути номери, обрати потрібний варіант і створити бронювання                   |
| Адміністратор | Переглядати заявки, керувати бронюваннями, змінювати статуси та працювати з номерами |

Клієнт використовує систему для створення заявки на проживання. Для нього важливо, щоб інтерфейс був зрозумілим, номери були представлені з описом і ціною, а форма бронювання мала чіткі поля та повідомлення про результат.

Адміністратор використовує систему для контролю бронювань. Для нього важливо бачити список заявок, дані клієнтів, номери, періоди проживання, статуси, суму та кнопки керування.

## Межі проєкту

Межі проєкту показують, які функції входять у поточну реалізацію програмної системи **«Готель»**.

До меж проєкту входить:

| Частина системи        | Що реалізовано                                           |
| ---------------------- | -------------------------------------------------------- |
| Головна сторінка       | Перегляд номерів, типів, цін, описів і статусів          |
| Сторінка бронювання    | Створення заявки клієнтом через форму                    |
| Перевірка даних        | Перевірка обов’язкових полів і коректності дат           |
| Адміністративна панель | Перегляд заявок, статусів, сум і статистики              |
| Керування бронюваннями | Підтвердження, заселення, скасування та видалення заявки |
| Керування номерами     | Додавання номера та редагування інформації про номер     |
| Сторінка вимог         | Таблиця покриття вимог реалізацією                       |
| Глосарій               | Пояснення основних термінів предметної області           |
| Серверна частина       | Обробка запитів і збереження даних                       |
| Збереження даних       | Збереження номерів і бронювань у JSON-файлах             |

Ці функції формують поточну межу навчального проєкту. Вони достатні для демонстрації основного сценарію роботи системи: клієнт переглядає номер, створює бронювання, система зберігає заявку, адміністратор бачить її та керує подальшим станом.

## Що не входить у межі поточної версії

Щоб межі проєкту були чіткими, потрібно також визначити функції, які не входять у поточну версію системи.

Поза межами поточної реалізації залишаються:

| Функція                     | Чому не входить у поточну версію                                                     |
| --------------------------- | ------------------------------------------------------------------------------------ |
| Онлайн-оплата               | Для навчального проєкту достатньо створення заявки без платіжної системи             |
| Авторизація користувачів    | Ролі клієнта й адміністратора показані через окремі сторінки, але без входу в акаунт |
| Реальна база даних          | Для поточного етапу використано JSON-файли                                           |
| Інтеграція з поштою або SMS | Система показує повідомлення в інтерфейсі, але не надсилає зовнішні сповіщення       |
| Мобільний застосунок        | Реалізовано вебінтерфейс у браузері                                                  |
| Багатоготельна мережа       | Проєкт орієнтований на одну програмну систему готелю                                 |
| Зовнішні сервіси бронювання | Система працює автономно без інтеграції зі сторонніми платформами                    |

Ці обмеження не є недоліком. Вони показують, що проєкт має визначені межі й не виходить за рамки навчального завдання.

## Реалізація в програмній системі «Готель»

У готовій програмі бачення продукту видно на головній сторінці. Там система представлена як вебзастосунок для перегляду номерів, створення бронювань, обліку заявок клієнтів і демонстрації покриття вимог. Користувач одразу бачить основне призначення системи й може перейти до ключових модулів.

Межі проєкту добре показані на діаграмі варіантів використання. У межах системи **«Програмна система “Готель”»** розміщені ті функції, які реально підтримуються програмою. Зовні показані актори — клієнт і адміністратор. Це дозволяє зрозуміти, хто взаємодіє із системою та які дії входять у межі поточної реалізації.

Для клієнта в межі проєкту входять перегляд номерів, перегляд інформації про номер, створення бронювання, перевірка доступності номера й отримання підтвердження бронювання.

Для адміністратора в межі проєкту входять перегляд списку бронювань, додавання номера, редагування інформації про номер, керування статусом заявки, підтвердження або скасування бронювання.

Цей скрін прямо підтверджує межі проєкту, тому що показує, які функції належать до системи, а які не відображені як частина поточної реалізації.

## Висновок

Отже, бачення продукту в проєкті **«Програмна система “Готель”»** полягає у створенні навчального вебзастосунку для роботи з номерами, бронюваннями, заявками, статусами, вимогами та глосарієм.

Межі проєкту визначають, що система охоплює перегляд номерів, створення бронювання, перевірку даних, збереження заявки, адміністрування бронювань, керування номерами, сторінку вимог і глосарій. Водночас онлайн-оплата, авторизація, зовнішні інтеграції, мобільний застосунок і багатоготельна мережа не входять у поточну версію.

Таким чином, проєкт має чітке бачення продукту та визначені межі, що дозволяє розробляти систему послідовно й не виходити за рамки поставленого навчального завдання.
