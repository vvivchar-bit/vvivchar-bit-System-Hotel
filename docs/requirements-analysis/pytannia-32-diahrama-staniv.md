# Питання 32. Діаграма станів

## Питання

**Діаграма станів.**

## Відповідь

Діаграма станів — це UML-діаграма, яка показує, у яких станах може перебувати об’єкт системи та як він переходить з одного стану в інший. Така діаграма використовується тоді, коли важливо описати життєвий цикл об’єкта: від початкового стану до завершення або зміни його статусу.

У програмній системі **«Готель»** діаграма станів потрібна для пояснення зміни стану номера або бронювання. Основними об’єктами, для яких важливі стани, є **номер готелю** та **заявка на бронювання**.

Номер може бути вільним, заброньованим або зайнятим. Бронювання може бути новою заявкою, підтвердженим, скасованим або видаленим. Саме ці переходи важливо показати на діаграмі станів, тому що вони пояснюють логіку роботи адміністратора та системи.

## Призначення діаграми станів

Діаграма станів допомагає зрозуміти, як змінюється об’єкт під час роботи системи.

У проєкті **«Готель»** вона пояснює:

| Що показує діаграма         | Значення для системи «Готель»                            |
| --------------------------- | -------------------------------------------------------- |
| Початковий стан             | Номер спочатку є вільним                                 |
| Перехід після бронювання    | Після створення заявки номер стає заброньованим          |
| Перехід після підтвердження | Адміністратор може підтвердити бронювання                |
| Перехід після заселення     | Після заселення номер переходить у стан зайнятого        |
| Скасування                  | Якщо заявку скасовано, номер знову може стати вільним    |
| Видалення                   | Якщо бронювання видалено, заявка прибирається із системи |

Таким чином, діаграма станів показує не просто перелік статусів, а логіку переходів між ними.

## Основні стани в системі «Готель»

У системі можна виділити такі основні стани:

| Стан          | Пояснення                                                       |
| ------------- | --------------------------------------------------------------- |
| Вільний       | Номер доступний для бронювання клієнтом                         |
| Нова заявка   | Клієнт створив бронювання, але адміністратор ще його не обробив |
| Заброньований | Номер прив’язаний до створеної або підтвердженої заявки         |
| Підтверджено  | Адміністратор підтвердив бронювання                             |
| Зайнятий      | Клієнт заселений, номер перебуває у використанні                |
| Скасовано     | Бронювання скасоване, номер може бути повернений у доступні     |
| Видалено      | Заявку прибрано зі списку бронювань                             |

Ці стани відповідають реальній логіці роботи готельної системи. Клієнт створює бронювання, а адміністратор керує подальшим станом заявки.

## Переходи між станами

У програмній системі **«Готель»** перехід між станами залежить від дій користувача або адміністратора.

| Перехід                                           | Що його викликає                               |
| ------------------------------------------------- | ---------------------------------------------- |
| Вільний → Нова заявка                             | Клієнт створює бронювання                      |
| Нова заявка → Підтверджено                        | Адміністратор підтверджує бронювання           |
| Підтверджено → Зайнятий                           | Адміністратор заселяє клієнта                  |
| Нова заявка → Скасовано                           | Адміністратор скасовує бронювання              |
| Підтверджено → Скасовано                          | Адміністратор скасовує підтверджене бронювання |
| Скасовано → Вільний                               | Номер знову може бути доступним для бронювання |
| Нова заявка / Підтверджено / Скасовано → Видалено | Адміністратор видаляє заявку                   |

Такі переходи показують, що стани в системі не змінюються випадково. Кожна зміна стану пов’язана з конкретною дією в програмі.

## Реалізація в програмній системі «Готель»

У готовій програмі стани проявляються через статуси номерів і бронювань.

Коли номер ще не заброньований, він має стан **«Вільний»**. Клієнт може обрати такий номер і створити бронювання. Після створення заявки система фіксує бронювання, а в адміністративній панелі з’являється новий запис зі статусом **«Нова заявка»**.

Після цього адміністратор може змінити стан заявки. Він може підтвердити бронювання, заселити клієнта, скасувати бронювання або видалити заявку. Кожна з цих дій змінює стан об’єкта.

Наприклад, якщо адміністратор підтверджує заявку, бронювання переходить зі стану **«Нова заявка»** у стан **«Підтверджено»**. Якщо адміністратор заселяє клієнта, стан змінюється на **«Зайнятий»**. Якщо бронювання скасовано, номер може повернутися до стану **«Вільний»**.

Таким чином, діаграма станів пояснює, як система керує життєвим циклом номера та бронювання.

## Значення діаграми станів для вимог

Діаграма станів важлива для аналізу вимог, тому що вона допомагає перевірити узгодженість статусів.

Наприклад, система не повинна одночасно вважати номер вільним і зайнятим. Так само заявка не може бути одночасно новою, підтвердженою і скасованою. Діаграма станів допомагає побачити допустимі переходи та уникнути суперечностей.

У системі **«Готель»** це пов’язано з вимогами до коректності й узгодженості. Якщо клієнт створив бронювання, воно повинно з’явитися в адміністративній панелі. Якщо адміністратор змінив статус, система повинна показати актуальний стан заявки.

## Висновок

Отже, діаграма станів є UML-моделлю, яка пояснює життєвий цикл об’єкта в програмній системі.

У проєкті **«Програмна система “Готель”»** вона показує, як змінюється стан номера або бронювання: від вільного номера до створення заявки, підтвердження, заселення, скасування або видалення.

Така діаграма допомагає перевірити коректність і несуперечність вимог, тому що вона показує допустимі стани та переходи між ними. Завдяки цьому можна зрозуміти, як система керує бронюваннями й статусами під час реальної роботи.
