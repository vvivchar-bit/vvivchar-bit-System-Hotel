# Питання 28. Специфікація функціональних вимог. Атрибути вимог

## Питання

**Специфікація функціональних вимог. Атрибути вимог.**

## Відповідь

Специфікація функціональних вимог — це структурований опис функцій, які повинна виконувати програмна система. Вона пояснює, які дії доступні користувачам, які дані обробляються, який результат має бути отриманий і як можна перевірити виконання кожної вимоги.

Функціональні вимоги описують саме поведінку системи. Вони відповідають на питання: **що система повинна робити?**

Атрибути вимог — це додаткові характеристики, які допомагають керувати вимогами, перевіряти їх реалізацію та відстежувати стан виконання. До таких атрибутів належать ID вимоги, формулювання, тип, пріоритет, реалізація, доказ перевірки та статус.

У проєкті **«Програмна система “Готель”»** функціональні вимоги описують основні можливості системи: перегляд номерів, створення бронювання, перевірку даних, збереження заявки, перегляд бронювань адміністратором, керування заявками та роботу зі статусами.

## Специфікація функціональних вимог у системі «Готель»

У системі **«Готель»** функціональні вимоги пов’язані з основними сценаріями роботи клієнта й адміністратора.

| ID    | Функціональна вимога                               | Реалізація в системі                                                                            |
| ----- | -------------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| FR-01 | Система повинна відображати список номерів готелю  | На головній сторінці показані картки номерів із типом, ціною, описом і статусом                 |
| FR-02 | Клієнт повинен мати можливість створити бронювання | На сторінці бронювання реалізована форма створення заявки                                       |
| FR-03 | Система повинна перевіряти обов’язкові поля        | Якщо користувач не ввів потрібні дані або вказав неправильні дати, система показує повідомлення |
| FR-04 | Система повинна зберігати бронювання               | Після створення заявки бронювання зберігається й доступне в адміністративній панелі             |
| FR-05 | Адміністратор повинен бачити список заявок         | В адмін-панелі є таблиця створених бронювань                                                    |
| FR-06 | Адміністратор повинен керувати бронюваннями        | Є кнопки підтвердження, заселення, скасування та видалення заявки                               |
| FR-07 | Кожен номер або заявка повинні мати статус         | У системі використовуються статуси номерів і бронювань                                          |

Ці вимоги є функціональними, тому що кожна з них описує конкретну дію або поведінку програмної системи.

## Атрибути вимог

Атрибути вимог потрібні для того, щоб вимоги були не просто текстом, а керованими елементами проєкту. Завдяки атрибутам можна зрозуміти, що це за вимога, де вона реалізована, як її перевірити та в якому вона стані.

У проєкті **«Готель»** для вимог використовуються такі атрибути:

| Атрибут вимоги     | Призначення                                             | Приклад у системі «Готель»                                   |
| ------------------ | ------------------------------------------------------- | ------------------------------------------------------------ |
| ID вимоги          | Унікальне позначення вимоги                             | `FR-01`, `FR-02`, `FR-03`                                    |
| Формулювання       | Опис того, що повинна виконувати система                | «Клієнт повинен мати можливість створити бронювання»         |
| Тип вимоги         | Показує, до якої групи належить вимога                  | Функціональна, нефункціональна, вимога до даних              |
| Реалізація         | Пояснює, де саме вимога виконана в програмі             | Форма бронювання, адмін-панель, головна сторінка             |
| Доказ перевірки    | Показує, яким скріном або дією можна підтвердити вимогу | Скрін форми бронювання або адмін-панелі                      |
| Статус             | Показує, чи виконана вимога                             | «Перекрито»                                                  |
| Пріоритет          | Визначає важливість вимоги для системи                  | Високий для створення бронювання й збереження заявки         |
| Критерій перевірки | Описує, як зрозуміти, що вимога виконана                | Після створення бронювання заявка з’являється в адмін-панелі |

## Реалізація в програмній системі «Готель»

У програмній системі **«Готель»** специфікація функціональних вимог реалізована через сторінку **«Покриття вимог реалізацією»**. На цій сторінці кожна вимога має ID, формулювання, опис реалізації, доказ для скріну та статус виконання.

Наприклад, вимога **FR-02** описує можливість створення бронювання. Вона реалізована на сторінці `booking.html`, де клієнт заповнює форму, обирає номер і створює заявку.

Вимога **FR-03** описує перевірку обов’язкових полів. Вона реалізована через повідомлення системи, які з’являються, якщо користувач не ввів потрібні дані або вказав некоректні дати.

Вимога **FR-05** описує можливість адміністратора бачити список заявок. Вона реалізована в адміністративній панелі, де створені бронювання відображаються у вигляді таблиці.

Вимога **FR-06** описує керування бронюваннями. Вона реалізована через кнопки дій: підтвердити, заселити, скасувати або видалити заявку.

Таким чином, кожна функціональна вимога має не лише текстове формулювання, а й конкретну реалізацію в інтерфейсі програми.

## Приклад атрибутів функціональної вимоги

| Атрибут      | Значення для вимоги FR-02                                |
| ------------ | -------------------------------------------------------- |
| ID           | FR-02                                                    |
| Назва        | Створення бронювання                                     |
| Формулювання | Клієнт повинен мати можливість створити бронювання       |
| Актор        | Клієнт                                                   |
| Реалізація   | Сторінка `booking.html`, форма бронювання                |
| Вхідні дані  | Ім’я, телефон, дата заїзду, дата виїзду, номер, коментар |
| Результат    | Створена заявка на бронювання                            |
| Доказ        | Скрін створення бронювання                               |
| Статус       | Перекрито                                                |
| Пріоритет    | Високий                                                  |

Цей приклад показує, що функціональна вимога повинна бути описана не тільки одним реченням. Для неї потрібно вказати, хто її використовує, де вона реалізована, які дані потрібні та який результат очікується.

## Значення атрибутів вимог для проєкту

Атрибути вимог допомагають організувати роботу над проєктом. Якщо вимога має ID, її легко знайти в таблиці. Якщо вказано реалізацію, зрозуміло, де саме вона виконана. Якщо є доказ, вимогу можна перевірити. Якщо є статус, видно, чи завершена її реалізація.

У проєкті **«Готель»** це особливо важливо, тому що одна вимога може бути пов’язана з кількома сторінками. Наприклад, створення бронювання починається на сторінці бронювання, але результат перевіряється в адміністративній панелі. Завдяки атрибутам можна показати повний зв’язок між вимогою, реалізацією та результатом.

## Висновок

Отже, специфікація функціональних вимог у проєкті **«Програмна система “Готель”»** описує основні дії, які повинна виконувати система: перегляд номерів, створення бронювання, перевірку даних, збереження заявки, перегляд бронювань і керування ними.

Атрибути вимог допомагають зробити ці вимоги зрозумілими, керованими та придатними до перевірки. У системі для вимог використовуються ID, формулювання, реалізація, доказ і статус виконання.

Таким чином, функціональні вимоги в проєкті **«Готель»** не залишаються загальними описами. Вони мають конкретну структуру, пов’язані з реальними сторінками програми та підтверджуються скрінами роботи системи.
