# Питання 26. Актори і варіанти використання

## Питання

**Актори і варіанти використання.**

## Відповідь

Актор у вимогах до програмної системи — це роль користувача або зовнішньої сутності, яка взаємодіє із системою для досягнення певної мети. Актор не обов’язково є конкретною людиною. Це саме роль у системі. Наприклад, одна людина може виконувати роль клієнта, якщо вона створює бронювання, або роль адміністратора, якщо вона керує заявками та номерами.

Варіант використання — це окрема дія або сценарій, який актор виконує в системі. Варіант використання показує, яку мету має актор і яку функцію система надає для досягнення цієї мети.

У проєкті **«Програмна система “Готель”»** основними акторами є **клієнт** і **адміністратор**. Клієнт використовує систему для перегляду доступних номерів, перегляду інформації про номер і створення бронювання. Адміністратор використовує систему для перегляду бронювань, підтвердження або скасування заявок, керування статусом заявки, а також для додавання та редагування інформації про номери.

## Основні актори системи

| Актор         | Опис ролі                                                        | Основна мета                                                          |
| ------------- | ---------------------------------------------------------------- | --------------------------------------------------------------------- |
| Клієнт        | Користувач, який переглядає номери готелю та створює бронювання  | Обрати номер і створити заявку на проживання                          |
| Адміністратор | Користувач, який керує бронюваннями, заявками та номерним фондом | Контролювати заявки, змінювати статуси, додавати та редагувати номери |

У системі **«Готель»** клієнт і адміністратор мають різні цілі. Клієнт працює з публічною частиною системи, де можна переглянути номери та створити бронювання. Адміністратор працює з керуючою частиною системи, де можна переглядати заявки, змінювати їх стан і керувати інформацією про номери.

## Варіанти використання для клієнта

Для актора **«Клієнт»** у системі визначено такі варіанти використання:

| Варіант використання              | Опис                                                                       |
| --------------------------------- | -------------------------------------------------------------------------- |
| Переглянути доступні номери       | Клієнт відкриває головну сторінку та бачить список номерів готелю          |
| Переглянути інформацію про номер  | Клієнт переглядає тип номера, опис, ціну та статус                         |
| Створити бронювання               | Клієнт заповнює форму бронювання та створює заявку                         |
| Перевірити доступність номера     | Система враховує стан номера перед створенням заявки                       |
| Отримати підтвердження бронювання | Після успішного створення заявки клієнт отримує повідомлення про результат |

Головним варіантом використання для клієнта є **«Створити бронювання»**, тому що саме цей сценарій реалізує основну цінність системи для користувача.

## Варіанти використання для адміністратора

Для актора **«Адміністратор»** у системі визначено такі варіанти використання:

| Варіант використання                 | Опис                                                                     |
| ------------------------------------ | ------------------------------------------------------------------------ |
| Переглянути список бронювань         | Адміністратор бачить усі створені заявки                                 |
| Підтвердити або скасувати бронювання | Адміністратор може змінити стан заявки після перевірки                   |
| Керувати статусом заявки             | Адміністратор може підтвердити, заселити, скасувати або видалити заявку  |
| Додати номер                         | Адміністратор може додати новий номер у систему                          |
| Редагувати інформацію про номер      | Адміністратор може змінити тип, ціну, опис, статус або зображення номера |

Після додавання вкладки **«Номери»** адміністратор у системі працює не тільки із заявками, а й із номерним фондом. Тому варіанти використання **«Додати номер»** і **«Редагувати інформацію про номер»** тепер відповідають реальній функціональності програми.

## Зв’язок акторів і варіантів використання

| Актор         | Варіант використання                 | Реалізація в програмі                                          |
| ------------- | ------------------------------------ | -------------------------------------------------------------- |
| Клієнт        | Переглянути доступні номери          | Головна сторінка з картками номерів                            |
| Клієнт        | Переглянути інформацію про номер     | Картка номера з типом, ціною, описом і статусом                |
| Клієнт        | Створити бронювання                  | Сторінка бронювання з формою введення даних                    |
| Клієнт        | Отримати підтвердження бронювання    | Повідомлення про успішне створення або помилку                 |
| Адміністратор | Переглянути список бронювань         | Адміністративна панель із таблицею заявок                      |
| Адміністратор | Підтвердити або скасувати бронювання | Кнопки дій у таблиці бронювань                                 |
| Адміністратор | Керувати статусом заявки             | Зміна стану заявки через кнопки адміністратора                 |
| Адміністратор | Додати номер                         | Вкладка **«Номери»**, форма додавання номера                   |
| Адміністратор | Редагувати інформацію про номер      | Вкладка **«Номери»**, кнопка **«Редагувати»** у списку номерів |

## Реалізація в програмній системі «Готель»

У програмній системі **«Готель»** актор **клієнт** починає роботу з перегляду номерів. Він бачить доступні номери, їх типи, ціни, описи та статуси. Після вибору номера клієнт переходить до сторінки бронювання, вводить дані та створює заявку.

Після успішного створення бронювання система показує повідомлення про результат. Це реалізує варіант використання **«Отримати підтвердження бронювання»**.

Актор **адміністратор** працює з адміністративною панеллю. У ній він переглядає список бронювань, бачить дані клієнта, номер, період проживання, статус заявки та кнопки керування. Адміністратор може підтвердити, заселити, скасувати або видалити бронювання.

Окремо в системі реалізовано вкладку **«Номери»**. Через неї адміністратор може додати новий номер або відредагувати інформацію про вже наявний номер. Це підтверджує, що варіанти використання, пов’язані з керуванням номерним фондом, реалізовані в програмі.

![Діаграма акторів і варіантів використання](screenshots/14-use-case-diagram.png)

На рисунку показано двох основних акторів системи: **клієнта** та **адміністратора**. Для клієнта визначено варіанти використання, пов’язані з переглядом номерів, створенням бронювання, перевіркою доступності номера та отриманням підтвердження. ## Висновок

Отже, у програмній системі **«Готель»** визначено двох основних акторів: клієнта та адміністратора. Клієнт використовує систему для перегляду номерів і створення бронювання. Адміністратор використовує систему для перегляду бронювань, керування статусами заявок, підтвердження або скасування бронювань, а також для додавання та редагування номерів.

Варіанти використання відповідають реальним сценаріям роботи програми. Вони підтверджуються UML-діаграмою та скрінами готової системи: створенням бронювання, адміністративною панеллю та сторінкою керування номерами.

Таким чином, актори і варіанти використання в проєкті **«Програмна система “Готель”»** визначені коректно, пов’язані з реальною функціональністю та підтверджені роботою програми.
