# Питання 17. Роль глосарію при аналізі вимог

## Питання

**Роль глосарію при аналізі вимог.**

## Відповідь

Глосарій при аналізі вимог — це впорядкований перелік основних термінів предметної області з поясненнями. Він потрібний для того, щоб усі учасники розроблення однаково розуміли ключові поняття системи.

Під час аналізу вимог одна й та сама назва може трактуватися по-різному. Наприклад, у проєкті **«Програмна система “Готель”»** терміни **«номер»**, **«бронювання»**, **«заявка»**, **«статус номера»**, **«статус бронювання»** мають конкретне значення. Якщо їх не пояснити, вимоги можуть стати неоднозначними.

Глосарій допомагає уникнути непорозумінь між розробником, користувачем, адміністратором, викладачем і перевіряючим. Він уточнює, що саме означає кожен термін у межах конкретного проєкту.

## Роль глосарію в аналізі вимог

Глосарій виконує кілька важливих функцій:

| Функція глосарію              | Значення для аналізу вимог                                               |
| ----------------------------- | ------------------------------------------------------------------------ |
| Уточнення термінів            | Допомагає зрозуміти, що саме означають основні поняття системи           |
| Зменшення неоднозначності     | Запобігає різному трактуванню однакових слів                             |
| Підтримка ясності вимог       | Робить формулювання вимог зрозумілішими                                  |
| Підтримка узгодженості        | Допомагає однаково використовувати терміни в інтерфейсі, вимогах і звіті |
| Полегшення перевірки          | Дає змогу перевіряючому зрозуміти, що саме реалізовано в системі         |
| Зв’язок із предметною областю | Пояснює, як терміни пов’язані з роботою готелю                           |

## Реалізація в програмній системі «Готель»

У проєкті **«Програмна система “Готель”»** глосарій реалізовано як окрему вкладку **«Глосарій»** у програмі. Це означає, що терміни не залишаються лише в тексті звіту, а є частиною самої системи.

На сторінці глосарію подано основні поняття, які використовуються під час аналізу вимог і розроблення системи:

* готель;
* номер;
* тип номера;
* клієнт;
* адміністратор;
* бронювання;
* заявка;
* дата заїзду;
* дата виїзду;
* статус номера;
* статус бронювання;
* адміністративна панель;
* вимога;
* покриття вимог.

Кожен термін має пояснення та значення для аналізу вимог. Це дозволяє зрозуміти, як саме поняття використовується в системі.

Наприклад, термін **«номер»** у глосарії означає не просто число, а окрему одиницю проживання в готелі, яка має номер кімнати, тип, ціну, опис і статус. Це важливо для правильного розуміння вимог до перегляду та бронювання номерів.

Термін **«бронювання»** означає заявку клієнта на проживання в певному номері у вибраний період. Завдяки цьому зрозуміло, що бронювання пов’язане з клієнтом, номером, датами проживання та статусом.

Термін **«заявка»** пояснює, як створене бронювання використовується адміністратором після його створення. Це допомагає пов’язати клієнтський сценарій бронювання з адміністративною панеллю.

| ----------------------------------------------------------- | --------------------------------------------------- |
| Система повинна відображати список номерів готелю           | Номер, тип номера, статус номера                    |

| Таблиця покриття вимог повинна показувати реалізацію вимог  | Вимога, покриття вимог, доказ, статус               |

## Чому глосарій важливий для системи «Готель»

Глосарій важливий для цього проєкту, тому що система працює з конкретною предметною областю — готелем. У цій області є власні поняття: номер, клієнт, бронювання, заявка, статус, дата заїзду, дата виїзду.

Якщо ці поняття не визначити, вимоги можуть бути нечіткими. Наприклад, вимога **«кожен номер повинен мати статус»** без глосарію може бути неповною, тому що незрозуміло, які саме статуси можливі. У глосарії визначено, що статус номера — це стан номера: **вільний, заброньований або зайнятий**.

Так само вимога **«адміністратор повинен бачити заявки»** стає зрозумілішою, якщо в глосарії пояснено, що заявка — це запис про створене бронювання, який відображається в адміністративній панелі.

Отже, глосарій допомагає зробити вимоги зрозумілими, узгодженими й придатними до перевірки.

## Підтвердження реалізації

Для цього питання використовується один прямий доказ — скрін окремої вкладки **«Глосарій»** у програмній системі **«Готель»**.

Цей скрін прямо підтверджує роль глосарію при аналізі вимог, тому що показує, що в програмі зафіксовано основні поняття предметної області: **готель**, **номер**, **тип номера**, **клієнт**, **адміністратор**, **бронювання**, **заявка**, **дата заїзду**, **дата виїзду**, **статус номера**, **статус бронювання**, **вимога** та **покриття вимог**.

Глосарій допомагає однаково трактувати ці поняття під час формування, реалізації та перевірки вимог.

## Висновок

Отже, глосарій відіграє важливу роль при аналізі вимог, тому що забезпечує єдине розуміння термінів предметної області та зменшує неоднозначність формулювань.

У проєкті **«Програмна система “Готель”»** глосарій реалізовано як окрему вкладку програми. Він пояснює основні поняття, які використовуються у вимогах, інтерфейсі та перевірці системи.

Завдяки глосарію вимоги стають більш ясними, узгодженими та придатними до перевірки, а розробник, користувач і перевіряючий однаково розуміють ключові терміни системи.
