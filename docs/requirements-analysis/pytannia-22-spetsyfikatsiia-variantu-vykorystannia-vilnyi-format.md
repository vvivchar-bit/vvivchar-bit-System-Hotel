# Питання 22. Специфікація варіанту використання. Вільний формат

## Питання

**Специфікація варіанту використання. Вільний формат.**

## Відповідь

Специфікація варіанту використання — це опис сценарію взаємодії користувача з програмною системою для досягнення конкретної мети. Вона показує, хто виконує дію, за яких умов вона починається, які кроки проходить користувач, як реагує система, які можуть бути альтернативні ситуації та який результат повинен бути отриманий.

Вільний формат специфікації означає, що опис не обов’язково має бути оформлений за жорстким стандартним шаблоном. Головне — щоб у ньому були зрозуміло описані актор, мета, передумови, основний сценарій, альтернативні сценарії, результат і зв’язок із вимогами системи.

У проєкті **«Програмна система “Готель”»** основним варіантом використання є **створення бронювання клієнтом**. Саме цей сценарій показує головну цінність системи: користувач обирає номер, вводить дані, створює заявку, а система зберігає бронювання та передає його в адміністративну панель.

## Специфікація варіанту використання: «Створення бронювання»

**Назва варіанту використання:** створення бронювання номера в готелі.

**Головний актор:** клієнт.

**Допоміжний актор:** адміністратор, який після створення заявки бачить її в адміністративній панелі та може обробити.

**Мета:** дати клієнту можливість створити заявку на проживання в обраному номері готелю.

**Передумови:**

* користувач відкрив програмну систему **«Готель»**;
* у системі є список номерів;
* користувач перейшов на сторінку бронювання;
* у формі доступні поля для введення даних;
* серверна частина працює та може зберегти заявку.

**Основний результат:** бронювання створено, збережено в системі та відображено в адміністративній панелі.

## Основний сценарій

| Крок | Дія користувача                                 | Реакція системи                                                 |
| ---- | ----------------------------------------------- | --------------------------------------------------------------- |
| 1    | Користувач відкриває сторінку бронювання        | Система показує форму створення заявки                          |
| 2    | Користувач вводить ім’я клієнта                 | Система приймає текстові дані                                   |
| 3    | Користувач вводить номер телефону               | Система приймає контактні дані                                  |
| 4    | Користувач обирає дату заїзду                   | Система фіксує початок періоду проживання                       |
| 5    | Користувач обирає дату виїзду                   | Система перевіряє, щоб дата виїзду була пізнішою за дату заїзду |
| 6    | Користувач обирає номер                         | Система прив’язує заявку до конкретного номера                  |
| 7    | Користувач натискає кнопку створення бронювання | Система перевіряє обов’язкові поля                              |
| 8    | Дані правильні                                  | Система створює бронювання та зберігає його                     |
| 9    | Бронювання створене                             | Система показує повідомлення про успішне створення              |
| 10   | Адміністратор відкриває адмін-панель            | Система показує створену заявку в таблиці                       |

## Альтернативний сценарій

Альтернативний сценарій виникає тоді, коли користувач вводить неправильні або неповні дані.

Наприклад, якщо користувач вказує дату виїзду раніше за дату заїзду, система не створює бронювання. Замість цього вона показує повідомлення:

**«Дата виїзду повинна бути пізнішою за дату заїзду.»**

Це означає, що система не дозволяє створити логічно неправильну заявку. Такий альтернативний сценарій потрібний для забезпечення коректності даних і несуперечності вимог.

## Постумови

Після успішного виконання варіанту використання в системі повинні бути виконані такі умови:

| Постумова                      | Пояснення                                                                       |
| ------------------------------ | ------------------------------------------------------------------------------- |
| Бронювання створене            | У системі з’являється нова заявка                                               |
| Дані збережені                 | Ім’я клієнта, телефон, номер, період проживання, статус і коментар зберігаються |
| Заявка доступна адміністратору | Адміністратор бачить бронювання в адміністративній панелі                       |
| Статистика оновлена            | Кількість бронювань збільшується, кількість вільних номерів зменшується         |
| Заявка має статус              | Нове бронювання отримує статус **«Нова заявка»**                                |

## Зв’язок із вимогами системи

Цей варіант використання пов’язаний з основними вимогами програмної системи **«Готель»**.

| Вимога                                             | Як реалізована у варіанті використання            |
| -------------------------------------------------- | ------------------------------------------------- |
| Клієнт повинен мати можливість створити бронювання | Реалізовано через форму бронювання                |
| Система повинна перевіряти обов’язкові поля        | Реалізовано через перевірку введених даних        |
| Дата виїзду повинна бути пізнішою за дату заїзду   | Реалізовано через повідомлення про помилку        |
| Система повинна зберігати бронювання               | Після створення заявка зберігається               |
| Адміністратор повинен бачити список заявок         | Створене бронювання відображається в адмін-панелі |
| Заявка повинна мати статус                         | Нове бронювання отримує статус **«Нова заявка»**  |

## Реалізація в програмній системі «Готель»

У готовій програмі цей варіант використання реалізовано через сторінку бронювання та адміністративну панель.

На сторінці бронювання клієнт вводить усі необхідні дані: ім’я, телефон, дату заїзду, дату виїзду, номер і коментар. Після натискання кнопки система перевіряє дані. Якщо всі умови виконані, створюється бронювання, а користувач отримує повідомлення про успішне збереження.

Якщо дані неправильні, система не створює заявку. Наприклад, коли дата виїзду вказана раніше за дату заїзду, система показує конкретне повідомлення про помилку. Це є альтернативним сценарієм варіанту використання.

Після успішного створення бронювання заявка з’являється в адміністративній панелі. Там адміністратор бачить клієнта, телефон, номер, період проживання, статус, суму та кнопки керування. Це підтверджує, що варіант використання завершується не лише повідомленням для клієнта, а й реальним збереженням заявки в системі.

## Підтвердження реалізації

Для цього питання використовуються три докази, які показують основний сценарій, альтернативний сценарій і результат виконання варіанту використання.

Цей скрін підтверджує основний сценарій варіанту використання: клієнт вводить дані й створює бронювання.

Цей скрін підтверджує, що специфікація варіанту використання враховує не лише успішний сценарій, а й помилкові ситуації.

Цей скрін підтверджує постумову варіанту використання: створене бронювання збережене в системі та доступне адміністратору для подальшої обробки.

## Висновок

Отже, специфікація варіанту використання у вільному форматі дозволяє описати реальний сценарій роботи користувача із системою без жорсткого шаблону, але з обов’язковим поясненням мети, актора, передумов, основного сценарію, альтернативних сценаріїв і результату.

У проєкті **«Програмна система “Готель”»** основним варіантом використання є створення бронювання. Цей сценарій охоплює введення даних клієнтом, перевірку правильності інформації, створення заявки, збереження бронювання та відображення його в адміністративній панелі.

Таким чином, варіант використання **«Створення бронювання»** пов’язує вимоги системи з реальною поведінкою програми та підтверджується конкретними скрінами основного сценарію, альтернативного сценарію й результату виконання.
