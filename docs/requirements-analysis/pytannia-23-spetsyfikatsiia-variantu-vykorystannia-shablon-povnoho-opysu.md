# Питання 23. Специфікація варіанту використання. Шаблон повного опису

## Питання

**Специфікація варіанту використання. Шаблон повного опису.**

## Відповідь

Специфікація варіанту використання у форматі повного опису — це детальний документований опис взаємодії користувача з програмною системою. На відміну від вільного формату, повний шаблон має чітку структуру: назву варіанту використання, актора, мету, передумови, основний сценарій, альтернативні сценарії, постумови, вимоги до даних і очікуваний результат.

Такий шаблон потрібний для того, щоб варіант використання був зрозумілим не лише розробнику, а й користувачу, тестувальнику, викладачу або іншому перевіряючому. Повний опис дозволяє побачити, хто виконує дію, за яких умов вона починається, які кроки проходить користувач, як реагує система та що має бути отримано після завершення сценарію.

У проєкті **«Програмна система “Готель”»** для повного опису обрано основний варіант використання **«Створення бронювання»**, тому що саме він є центральним сценарієм роботи системи.

## Передумови

Перед початком виконання варіанту використання мають бути виконані такі умови:

| Передумова                   | Пояснення                                            |
| ---------------------------- | ---------------------------------------------------- |
| Система запущена             | Користувач відкриває програму через браузер          |
| Доступна сторінка бронювання | Користувач може перейти на сторінку створення заявки |
| У системі є номери           | Користувач має з чого обрати номер                   |
| Форма бронювання доступна    | Користувач бачить поля для введення даних            |
| Серверна частина працює      | Система може зберегти створену заявку                |

## Альтернативні сценарії

| Альтернативний сценарій           | Умова виникнення                                 | Реакція системи                                                                   |
| --------------------------------- | ------------------------------------------------ | --------------------------------------------------------------------------------- |
| A1. Не заповнено обов’язкове поле | Користувач не ввів ім’я, телефон, дату або номер | Система не створює бронювання та показує повідомлення про помилку                 |
| A2. Некоректні дати               | Дата виїзду раніше або не пізніше дати заїзду    | Система показує повідомлення: «Дата виїзду повинна бути пізнішою за дату заїзду.» |
| A3. Не обрано номер               | Користувач не вибрав номер зі списку             | Система не створює заявку й вимагає обрати номер                                  |
| A4. Дані не збережено             | Помилка під час обробки даних                    | Система повинна повідомити користувача про проблему збереження                    |

## Постумови

Після успішного виконання варіанту використання мають бути виконані такі умови:

| Постумова                      | Пояснення                                                               |
| ------------------------------ | ----------------------------------------------------------------------- |
| Бронювання створено            | У системі з’являється нова заявка                                       |
| Дані клієнта збережено         | Зберігаються ім’я, телефон, номер, дати проживання, статус і коментар   |
| Заявка доступна адміністратору | Адміністратор бачить бронювання в адміністративній панелі               |
| Номер отримує відповідний стан | Система враховує, що номер уже використовується в бронюванні            |
| Статистика оновлена            | Кількість бронювань збільшується, кількість вільних номерів зменшується |
| Заявка має статус              | Нове бронювання отримує статус **«Нова заявка»**                        |

## Реалізація в програмній системі «Готель»

У програмній системі **«Готель»** цей варіант використання реалізовано через сторінку бронювання та адміністративну панель.

Клієнт відкриває сторінку бронювання, вводить ім’я, телефон, дату заїзду, дату виїзду, обирає номер і натискає кнопку створення бронювання. Якщо всі дані введені правильно, система створює заявку та показує повідомлення про успішне збереження.

Якщо дані введені неправильно, система не створює бронювання. Наприклад, якщо дата виїзду раніше дати заїзду, система показує повідомлення про помилку. Це відповідає альтернативному сценарію повного опису варіанту використання.

Після успішного створення заявки адміністратор бачить її в адміністративній панелі. У таблиці відображаються клієнт, телефон, номер, період проживання, статус і кнопки керування. Це підтверджує постумови варіанту використання.

## Підтвердження реалізації

Для цього питання використовуються три докази, які прямо відповідають структурі повного опису варіанту використання: основний сценарій, альтернативний сценарій і постумови.

### Рисунок 1 — Основний успішний сценарій створення бронювання

![Створення бронювання](screenshots/02-booking-created.png)

На рисунку показано успішне створення бронювання. Користувач заповнив форму, система прийняла дані та показала повідомлення **«Бронювання успішно створено та збережено.»**

Цей скрін підтверджує основний сценарій повного опису варіанту використання.

### Рисунок 2 — Альтернативний сценарій із некоректною датою

![Перевірка коректності дат](screenshots/10-correctness-date-validation.png)

На рисунку показано альтернативний сценарій. Користувач ввів дату виїзду, яка є ранішою за дату заїзду. Система не створила бронювання та показала повідомлення **«Дата виїзду повинна бути пізнішою за дату заїзду.»**

Цей скрін підтверджує, що повний опис варіанту використання враховує не тільки успішний сценарій, а й помилкові ситуації.

### Рисунок 3 — Постумова: створене бронювання доступне адміністратору

![Адміністративна панель зі створеним бронюванням](screenshots/11-consistency-booking-status.png)

На рисунку показано адміністративну панель після створення бронювання. У таблиці відображено заявку клієнта, телефон, номер, період проживання, статус **«Нова заявка»** і кнопки керування.

Цей скрін підтверджує постумову варіанту використання: бронювання збережене в системі та доступне адміністратору для подальшої роботи.

## Висновок

Отже, специфікація варіанту використання за шаблоном повного опису дозволяє детально описати взаємодію користувача з програмною системою. Вона містить назву, актора, мету, передумови, основний сценарій, альтернативні сценарії, постумови, спеціальні вимоги та зв’язок із вимогами системи.

У проєкті **«Програмна система “Готель”»** повний опис варіанту використання **«Створення бронювання»** показує, як клієнт створює заявку, як система перевіряє дані, як обробляє помилки та як результат відображається в адміністративній панелі.

Таким чином, повний опис варіанту використання допомагає пов’язати вимоги з реальною поведінкою програми та зробити сценарій зрозумілим для розробника, користувача, адміністратора й перевіряючого.
